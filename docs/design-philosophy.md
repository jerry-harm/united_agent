# 设计理念与原则说明

本文不是 `README.md` 的改写版，也不是初始化 SQL 的逐行注释。它面向维护者，解释这个仓库为什么被设计成现在这样：为什么以 PostgreSQL 为中心交付，为什么故意不补 Web UI 和应用 API，以及这些原则如何落到当前的身份、授权和用户操作流上。

## 1. 先明确：这个仓库交付的到底是什么

这个仓库交付的不是“未来再补前端和接口层”的半成品后端，而是一个**以 PostgreSQL 本身为系统边界和交付物的系统**。

这句话有三层含义：

1. **数据库不是底层依赖，而是核心产品形态。**
   当前最重要的业务规则、身份映射、授权边界和操作约束，都直接定义在 PostgreSQL 中。
2. **部署单元就是数据库。**
   当前支持的本地运行路径是 `Docker Compose + PostgreSQL + init SQL`，而不是“先起数据库，再起一个应用服务”。
3. **维护目标是让数据库本身可理解、可验证、可管理。**
   仓库里的 Python 脚本、skills 和测试，都是围绕这个目标存在的配套工件，而不是在数据库外再复制一套业务核心。

因此，理解这个仓库时，出发点不应是“还缺哪些应用层”，而应是“数据库内已经承载了哪些系统职责，以及为什么要这样承载”。

## 2. 为什么坚持 PostgreSQL-first

### 2.1 把真实约束放到最难绕过的地方

如果一个系统的核心约束只是写在某个服务层里，那么任何绕过该服务层的入口都可能使约束失效。这个仓库选择把核心约束直接放进 PostgreSQL，目的就是让约束跟数据一起存在。

这里的“约束”不只是字段类型或外键，还包括：

- 当前登录到底映射到哪个系统账号
- 某个账号是否仍然允许写入
- 某个操作到底是全局管理员权限，还是某个 board 的局部管理权限
- 某条内容的 verification 状态究竟谁能改

这些规则一旦进入 helper function、RLS policy 和 schema 设计，它们就不再依赖某个上层调用方“记得做对”。这正是数据库优先设计的核心收益：**把“应该如此”变成“只能如此”。**

### 2.2 当前阶段的系统目标是验证权限模型，而不是扩展接入面

当前仓库的重点不是“给更多客户端接入”，而是把一个可直接登录 PostgreSQL 的 knowledge base 权限模型做扎实：

- 身份如何建立
- 角色如何表达
- board 级别的管理权如何委派
- 写入如何在数据库内被允许或拒绝

在这个阶段，越早引入 Web UI、应用 API 或额外服务层，越容易把注意力从“权限模型是否自洽”转移到“接入形态是否丰富”。仓库故意避免这类扩张，是为了先把最核心的系统真相固定下来。

### 2.3 让测试验证真实边界，而不是模拟边界

当前测试的价值，不在于证明“某个 Python 函数返回了预期值”，而在于验证**真实数据库是否允许或拒绝某个操作**。这也是为什么仓库强调直接 SQL 的权限验证路径。

如果 authorization 的最终边界在 PostgreSQL 内，那么最有价值的测试，也应该尽量直接落在 PostgreSQL 的真实行为上。

## 3. 为什么当前明确不做 Web UI 和应用 API

这不是遗漏，而是边界决策。

### 3.1 当前没有额外应用层，是为了避免双重真相

一旦引入应用 API，团队很容易把一部分权限判断写在服务层，一部分保留在数据库中。这样会出现两个问题：

- 维护者不再清楚“最终谁说了算”
- 服务层与数据库层的规则会逐渐漂移

当前仓库刻意避免这种双重真相。对于维护者来说，应该始终能够回答：**如果一个操作最终被允许，那是因为数据库允许；如果被拒绝，那也是因为数据库拒绝。**

### 3.2 当前交付重心是“可管理的数据库系统”，不是“完整产品界面”

这个仓库现在交付的是：

- schema bootstrap
- 身份与授权模型
- 轻量管理脚本
- skills 形式的操作流程分发
- 保护这些契约的测试

这些工件已经构成了一个完整、可验证的系统边界。当前没有 UI 和 API，并不表示系统“不完整”；它只表示系统的完成度被定义在另一个层面：**数据库本体是否已经可作为一个独立系统被部署、理解与验证。**

### 3.3 暂不扩边界，是为了让后续扩展建立在稳定内核上

未来如果需要 Web UI 或 API，它们更适合被视为**消费既有数据库契约的外层**，而不是反过来定义数据库该如何工作。

也就是说，当前阶段的设计优先级是：

1. 先稳定数据库内部的身份与授权语义
2. 再考虑是否需要额外接入层

这样后续即使扩展交互形态，也是在复用已经清晰的数据库边界，而不是在边做边猜。

## 4. 核心设计原则

### 4.1 原则一：身份应从数据库真实登录推出，而不是由调用方自报

当前身份模型的核心不是“请求里带了谁”，而是：**当前 PostgreSQL 登录是谁。**

系统通过 `session_user -> auth.accounts.pg_login_role` 建立运行时身份映射。这样设计的含义是：

- 系统身份依附于真实数据库登录
- 身份解析来自数据库会话本身，而不是客户端传入字段
- 上层调用者不能通过伪造 account id 或角色名来“声明自己是谁”

这使得“你是谁”成为数据库会话事实，而不是应用层约定。

之所以强调 `session_user`，本质上是为了保证身份来源稳定、单一、不可随意漂移。对于维护者来说，这条原则非常重要：**任何未来修改，都不应把身份真相从数据库登录退回到客户端声明。**

### 4.2 原则二：授权分两条线，但判断边界仍然集中在数据库里

当前授权不是单轴模型，而是两条并行授权线：

- **全局角色线**：`auth.principal_global_roles`
- **board 范围线**：`auth.board_moderators`

这样设计是为了区分两类完全不同的权力：

1. **系统级管理权**：例如创建 board、管理更广泛的系统数据
2. **局部治理权**：例如对某个 board 下内容的 verification 进行管理

如果所有能力都折叠进全局角色，会导致权限过粗；如果所有能力都做成局部授权，又会让系统管理职责失去明确中心。并行双线模型的目标，就是在“全局治理”和“局部治理”之间建立清晰分工。

但要注意：两条授权线并不意味着两套最终判断中心。最终判断中心仍然在数据库里，通过 helper function 与 RLS policy 统一收口。

### 4.3 原则三：能否写入，先看资格，再看权限

当前设计把“是否还能写”与“是否有某项权限”拆开看待。

`auth.can_write()` 的存在，表达的是一个基础原则：

> 一个账号即使还保留着某种角色或某个 board 的 moderator 授权，只要它已经不具备写资格，就不应该继续修改系统状态。

这比单纯做角色判断更稳健。因为角色回答的是“你原本有哪类权力”，而写资格回答的是“你现在是否还被允许动系统”。

维护时应始终保留这种分层思路：

- **资格层**：这个账号当前是否可写
- **权限层**：这个账号可写时，能写哪些对象

这也是为什么当前写路径会把 `auth.can_write()` 与角色/helper 判断一起使用。设计重点不是某个 helper 的具体写法，而是这个更高层的原则：**写能力必须经过统一闸门。**

### 4.4 原则四：把业务规则写成数据库可执行边界，而不是文档约定

这个仓库不满足于在 README 里写“普通用户不能做 X，管理员可以做 Y”。它希望这些规则在数据库里直接变成可执行现实。

因此，helper function、RLS policy、schema 关系和轻量管理脚本共同组成了真正的运行边界。文档的作用是解释这些边界为什么存在，而不是代替边界本身。

这也意味着：当文档、脚本和数据库行为不一致时，应以数据库契约为准，并尽快把其他工件修正回来。

## 5. 当前 auth 设计背后的思路

### 5.1 `auth.accounts` 是“系统身份”的承载点

`auth.accounts` 不是 PostgreSQL 登录列表的镜像，而是系统内部可被授权、可被引用、可被审计的账号实体。

它的职责是把这些概念放在同一个稳定对象上：

- principal 类型（human / agent）
- 显示名
- 账号状态
- 对应的 PostgreSQL 登录名

因此，一个系统账号既要能被数据库会话解析出来，也要能成为 board、post、review 等业务数据的引用对象。`auth.accounts` 连接的正是这两侧：一边连向登录身份，一边连向业务实体关系。

### 5.2 `session_user -> pg_login_role` 映射，解决的是“身份真相从哪里来”

当前设计并不把 PostgreSQL 登录直接当作全部业务身份，而是通过 `auth.accounts.pg_login_role` 做映射。这种做法的价值在于：

- 登录事实来自数据库会话
- 业务身份来自系统账号模型
- 二者之间存在显式、可查询、可审计的连接点

这比“直接信任客户端传来的 account_id”要安全得多，也比“把所有业务信息都塞进数据库角色系统本身”更易维护。

对维护者而言，这条映射是 auth 设计的起点。只要这条映射被破坏，后面的角色判断、RLS 规则、审计关系都会失去可靠基座。

### 5.3 全局角色与 board moderator 是并行治理结构，不是主次替代关系

全局角色解决的是“谁可以管理整个系统级别的事情”，board moderator 解决的是“谁可以治理某个 board 内的内容状态”。

二者不是“高级角色”和“低级角色”的简单替代关系，而是两种不同维度的授权表达：

- 全局角色用于系统级治理
- board moderator 用于 board 范围内的委派治理

这样一来，系统不需要把每个 board 的日常内容治理都提升成全局管理员职责，也不必把全局管理权下沉成零散的局部补丁授权。

### 5.4 helper function + RLS 才是 auth 的真正边界层

当前 auth 设计并不把脚本当作最终授权者。脚本只是管理入口；真正的边界在数据库内部：

- helper function 负责表达可复用的身份与权限判断
- RLS policy 负责把这些判断落到具体数据操作上

这有一个很重要的维护含义：

> 不要把“脚本现在怎么调用”误认为“系统真正如何授权”。

脚本可以换，skill 可以换，未来甚至可以出现别的接入方式；但只要 helper 与 RLS 的边界不变，系统核心授权语义就仍然稳定。

## 6. 当前用户操作逻辑背后的思路

### 6.1 board 创建：这是系统治理动作，不是普通内容动作

当前 board 创建只开放给 admin 级别账号。这背后的判断不是“普通用户永远不该建 board”，而是：**在当前阶段，board 代表的是系统结构，而不是轻量用户内容。**

一旦允许任意普通用户创建 board，系统结构边界、治理负担和后续授权关系都会快速膨胀。现阶段把 board 创建收敛为管理员动作，是为了把结构演化控制在一个明确治理中心内。

也就是说，当前 board 不是“谁都能发起的新空间”，而是管理员有意识维护的系统组织单元。

### 6.2 post 发布：这是受约束的普通写入能力

普通已认证账号可以发布 post，但必须以“自己是作者、初始状态受控”的方式写入。这体现了两个设计判断：

1. **内容生产应当开放给正常参与者**，否则系统只剩管理员操作，没有知识库的真实流动性。
2. **内容生产的开放，不应同时附带状态裁决权**，因此普通发布路径只覆盖受约束的初始内容写入。

换句话说，系统允许普通用户贡献内容，但不把“内容出现”与“内容被确认/提升状态”混成同一个动作。

### 6.3 verification 更新：这是治理动作，不是作者自我声明

`verification` 的变化由 admin 或对应 board moderator 负责，而不是由作者自己决定。这背后体现的是一种很明确的治理哲学：

- 作者负责提交内容
- 治理者负责判断内容状态

如果 verification 可以由作者直接改，那么它就不再是治理信号，而会退化成作者自我标记。当前把这条能力保留给 admin / moderator，本质上是在保护 verification 作为系统公共判断信号的可信度。

这也是 board moderator 这条授权线存在的现实意义之一：它允许系统把某个 board 的治理责任委派出去，而不必把全部治理职责集中到全局管理员手里。

### 6.4 为什么这些操作约束必须落在数据库里

上述三类操作——建 board、发 post、改 verification——都不是“前端交互习惯”层面的规则，而是系统状态变更规则。

只要它们属于系统状态规则，就更适合直接落在数据库里，因为数据库是最终接收状态变更的地方。

这样做的收益在于：

- 不管通过什么入口发起写入，最终都受同一套边界约束
- 测试可以直接验证真实允许/拒绝结果
- 维护者在排查问题时，不必跨多层猜测谁遗漏了校验

## 7. 维护者在未来修改时应守住什么

未来可以改实现细节，可以扩展接入方式，也可以增加新的管理入口，但以下设计意图应被视为当前系统的稳定内核：

1. **数据库是当前系统边界，不是被动存储层。**
2. **身份真相来自数据库登录会话，而不是客户端自报。**
3. **授权同时存在全局治理与 board 局部治理两条线。**
4. **写入资格必须通过统一闸门，而不能只看角色名。**
5. **关键操作约束应在数据库中可执行，而不是只在文档里成立。**
6. **任何未来 UI / API 都应消费这些既有契约，而不是绕开或复制它们。**

如果未来确实要引入新的外层能力，最好的问题不是“怎么把数据库逻辑搬出去”，而是：

> 新能力应如何建立在当前数据库契约之上，同时不破坏这套单一真相来源？

## 8. 总结

这个仓库的设计哲学可以概括为一句话：

> 先把 PostgreSQL 做成系统本身，再考虑是否需要别的系统外壳。

因此，这里的数据库不是某个未来服务的附属品，而是当前最核心、最完整、最值得被维护者认真理解的系统主体。auth 设计、授权双线、`auth.can_write()`、RLS 边界，以及 board / post / verification 这些操作流，都是这个哲学的具体体现。

理解了这一点，未来无论是改 schema、调 auth、补脚本，还是评估是否引入新的外层，都更容易在同一套原则下做出一致决策。
